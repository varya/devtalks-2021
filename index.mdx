## Measuring production costs
{ .slide--shout }

### to reflect, how easier (and more cost-effective) the life is for the projects

<Stamp title="past"/>

<!--
If we analyse our past, we can measure many things. One of the most obvious
metrics would be production costs. In short, how much money a design system
saves us.
-->


## Costs and savings w/o DS

### 1 component, 1 project

```
$ = production time + (support costs * component lifetime)
```

### same component, many projects
```
$ = (
    production time + (support costs * component lifetime)
  ) * amount of projects
```

## Costs and savings /w DS

### 1 component, in the design system

```
$ = production time + (support costs * component lifetime)
```

### same component, at the projects
```
$ = integration costs * amount of projects
```

### altogether
```
$ = $(design system) + $(projects)
```


<!--
Re-using of the components definetely saves us design and development
time spent. But we cannot just multiply how many times the component
was used to its one-time costs and say that we saved that much money.

Using a design system costs too. We can work on decreasing these costs
but anyway we should say out load that linking a library of components
to the project also requires some work. Reading documentation costs time.
Possible bugs, repoting them, getting updates with fixes â€” that all forms
integration costs.

From design system team, they have to support. So, the more they are offering,
the more components they serve, the more is the support load.

But luckily we support only one component, thanks to the reuse.

-->

## Where to get this data from?

`production time` â€” survey, tracker, documented time

`support costs` - estimation

`integration costs` - survey, tracker, documented, estimation

`component lifetime` - git history, data from design files

`amount of projects` - github search, figma data, survey

<!--
How exactly a team gets the data depends on many factors. Mostly,
on the shape of their organization and the tools they are using.

If code is stored on in some repository, say GitHub, it is possible to
obtain the data via search. In my recent project, we were using
a custom script that crawls across company's GitHub Enterprise and gets
all the needed data.

With Figma it is a bit more tricky. On Eternprise licenses there is
some statistics but if we need per component, it is not possible to
export automatically. Probably it can be done via Figma API but I haven't
checked. I consider it as a minor problem because these stats snapshots
are not needed every day. Once a month is already quite often, and we can
just manually re-write the numbers into some file where we store it.

You can always go with a simpliest version â€” just to run a survey, so that
people tell you what kind of components they use. This is already some data.
-->


## Example: button component

```
production time = 10 h         integration costs = 2h
support costs = 20 h/year      amount of projects = 5
component lifetime = 3 years
```

### costs and savings

```
$(without ds) = (10h + 20h*3y)*5 = 350h
$(with ds) = (10h + 20h*3y) + 2h*5 = 80h

SAVED ON THE BUTTON FOR 3 YEARS: 350h - 80h = 270h
```

## How accurate is that?

The formulas don't have to be perfect.

They help to <u>start a conversation</u>
with the business.

Together, you will co-create better formulas for your
paricular organisation.
